# L4 · Query 优化：在检索之前修「提问本身」

> 实验：`experiments/rag-query-opt/query_opt_demo.py`（纯标准库，四路对照全绿，MRR 0.458 → 1.000）

## 这一层修的是哪个环节

RAG 出错，未必是检索器差，可能是**用户的提问本身**不适合检索：

```
用户口语提问 ──▶ [L4 Query 优化] ──▶ 检索器(L2 Hybrid) ──▶ 精排(L3 Rerank) ──▶ LLM
                    ↑ 本层
```

- **L2 Hybrid** 修「召回通道」：向量漏关键词，补 BM25。
- **L3 Rerank** 修「候选内的序」：粗召回排序噪声，用 Cross 交互精排。
- **L4 Query 优化** 修「提问本身」：口语化、词表错配、缺过滤条件——发生在**检索之前**。

一句话边界：**L4 改输入，L2 改召回，L3 改排序。三者正交，可叠加。**

## 为什么需要：查询侧的三类硬伤

| 硬伤 | 例子 | 后果 |
|------|------|------|
| 词表错配（vocabulary mismatch） | 用户说「白拿几天不上班的假」，制度写「带薪年假」 | 字面零重合，BM25 失效；短口语向量也偏 |
| 问题过短/语域偏 | 「年假几天？」vs 答案型长文档 | 短问题的字符分布离「文档」远，稠密检索也偏 |
| 缺过滤条件 | 「年假余额几天」——FAQ 噪声页也含「年假/余额」 | 跨库噪声抢字面，gold 被压到 @3 |

实验里 raw hybrid 在四道口语题上分别命中 @2 / @2 / @2 / @3，**没有一道 top1**。问题不在检索器，在 query。

## 四种方法（各修一类硬伤）

### 1. Query Rewrite — 归一到规范术语
把口语提问改写成检索友好的表达（保留原问 + 追加术语，避免丢信息）。
- 生产等价物：LLM「请把用户问题改写为检索友好的规范表达」。
- 实验：`老员工白拿几天不上班的假？` → 追加 `带薪 年假 工龄满 天数`，gold 从 @2 翻到 top1。
- 治：**词表错配**。

### 2. HyDE（Hypothetical Document Embeddings）— 先答后检索
先让模型编一段**假想答案**，用假答案去检索。假答案的语域比短问题更贴近真实文档。
- 生产等价物：LLM「先写一段可能的答案」→ embed 假答案检索。
- 实验：`蹭公共网络要不要开那个翻墙工具？` → 假答案 `公共 Wi-Fi 下必须开启公司 VPN…`，把 VPN 制度页从 @2 拉回 top1。
- 治：**问题过短/语域偏**。代价：假答案可能带幻觉，命不中意图要能退回原问。

### 3. Multi-Query — 一问多改写 + RRF 融合
把一个问题改写成多个视角（原问 + rewrite + HyDE），各自检索，再 RRF 融合，扩大召回面、降低单一问法的偶然性。
- 生产等价物：LLM「生成 3 个语义等价但措辞不同的检索式」。
- 实验：`老员工出差报多少` raw 被年假页抢 top1（@2），三变体并检 + RRF 把差旅正文顶回 top1。
- 治：**单一问法召回窄/偶然漂序**。代价：N 倍检索开销。

### 4. Self-Query — 抽结构化过滤条件
从问题里抽出 metadata 过滤条件（topic / 时间 / 作者），检索前先过滤，压制跨库噪声。
- 生产等价物：LLM「从问题里抽出 where 条件」→ 向量库带 metadata filter 检索。
- 实验：`年假余额到底几天？` 抽出 `topic=hr-leave`，把 FAQ 噪声页（同含「年假/余额」）挡在候选外，gold 从 @3 翻到 top1。
- 治：**缺过滤条件/跨库噪声**。代价：抽错过滤条件会误杀 gold（宁可放宽不要漏召）。

## 边界与选型

- **不是每题都要上**：字面清晰的专名题（`X-KEY-99`）raw 已 top1，加改写是浪费 token。先判「query 是否有硬伤」再决定上哪种。
- **延迟/成本**：Rewrite/HyDE ≈ +1 次 LLM 调用；Multi-Query ≈ +1 次 LLM + N 倍检索。高 QPS 场景要算 ROI。
- **HyDE 幻觉**：假答案错了会把检索带偏，务必保留「命不中意图退回原问」的兜底。
- **Self-Query 误杀**：抽错过滤条件比不过滤更糟——过滤规则要保守，抽不准就不过滤。
- **与 L3 的分工**：Query 优化改「进什么候选」，Rerank 改「候选怎么排」。Query 侧能召回的，Rerank 才有的排；召不回的，Rerank 也救不回（见 L3 铁律）。
